Vercel Serverless API Routes (linkd-fms/api/)

RouteModesUsed ByService-role action
POST /api/admin-update-user list_emails / fetch / update TeamView (Email column + Edit dialog) Reads auth.users email + updates email/password/profile fields (full_name, role, is_active, created_at)
Auth flow: caller's JWT (Authorization: Bearer ...) is verified server-side; only admin / design_coordinator profiles are accepted.
Env (Vercel only): SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY. The service-role key NEVER ships to the browser.
Client helper: callAdminApi(route, body) in src/lib/adminApi.ts attaches the Bearer token and returns {data, error}.
Local dev: npm run dev doesn't serve these (Vite only). Use npx vercel dev to test the API locally; otherwise the Team email column shows a friendly "needs vercel dev" note in development.
Convention: new privileged endpoints go here, NOT into Supabase Edge Functions. The two existing edge functions (admin-create-user, daily-notifications) stay on Supabase for historical reasons.

3. Role Permissions (5 Roles)

Super Admin Highest Level

  • All Admin powers + Danger Zone access
  • Stage 2 concept approval (MD review)
  • Full system configuration
  • User role management + designer codes
  • Scorecards (grid + detail views)
  • CSV Export on all views
  • Sampling, Salvedge, Files, Team

Admin Full Power

  • All Tasks, Task Dashboard, Concept Dashboard
  • Stage 2 concept approval (admin/super_admin only)
  • Change user roles + manage designer codes
  • Manage lookup tables (System CRUD)
  • Scorecards (grid + detail views)
  • CSV Export on all views
  • Sampling, Salvedge, Files, Team

Coordinator Admin-Equivalent

  • Same as Admin (permissions now equal)
  • Task lifecycle (create, assign, advance)
  • Coordinator Tasks page (/coordinator-tasks)
  • Stage 4 final approval access (NOT Stage 2)
  • User management (add/edit/deactivate)
  • System CRUD access
  • Sampling, Salvedge, Files, Team
  • No Scorecards page access

Designer Submit & Work

  • My Board, Task Dashboard, Concepts, Files
  • Claim tasks from open pool
  • Submit + finalize concepts
  • Can edit own concepts until MD approves
  • Hold / Resume concepts
  • Own scorecard view (/scorecards/:self)
  • Salvedge (own assigned records only)
  • Cannot create briefs or manage team
  • No Sampling, System access

DEO Kitting Only

  • Kitting Queue (/kitting) is their landing page
  • Digitizes paper kitting forms (12-field)
  • Can claim + complete kitting records
  • Profile + Notifications only
  • Cannot create tasks or briefs
  • No dashboards, concepts, files, team

4. Task Pipeline

Pool
Unassigned · FIFO claim
In Progress
Designer working
Done
Awaiting fabric
Completed
Terminal (fabric recorded)
Full Kitting
Side data tab (not a pipeline stage) · 2-stage coordinator→DEO workflow on the row's action menu
Sampling
Own page at /sampling · runs in parallel, not a downstream stage
Stepper UI: <TaskPipelineStepper> — slim single-row glass pills mounted as the All Tasks table header. Connected: Pool → In Progress → Completed (chevrons fill once upstream has items). Full Kitting renders as a standalone side pill divided off to the right (NOT a pipeline step) via the sideStage prop.
done vs completed (load-bearing): done = design work finished, fabric not yet captured (intermediate "awaiting completion details" state). completed = terminal once fabric is recorded. On dashboards, "Done" merges both; on the board, done tasks stay in In Progress badged "Done" with a per-row "Complete" CTA.
Pool claim flow: Designers don't browse the pool — <ClaimTaskModal> shows only the single FIFO-eligible task (urgent-first, then oldest requirement_received_at) with all brief details + reference files. They claim that one or cancel — no cherry-picking. Claim sets planned deadline (required), optional fabric, status=in_progress, designer letter on task_code, optimistic lock on .eq('status','pool').
Completion: Mark Done → done + notifies designer + admins/coordinators. "Complete" then runs immediately if fabric was set at claim time, otherwise opens <PostDoneModal> for the required fabric. completeTask stamps completion_fabric/filled_by/filled_at (migration 0040). MTR was removed.
qty_completed may EXCEED qty: the inline column CHECK referenced two columns and Postgres auto-named it tasks_check (NOT tasks_qty_completed_check). 0043 dropped the wrong name; 0044 drops the real one and re-asserts only qty_completed >= 0. QtyTracker clamp has no upper bound.
Realtime: tasks joined the supabase_realtime publication (0041, REPLICA IDENTITY FULL); pool claims/assignments propagate live (~1s). concepts followed (0054).
Task code: DF NN-D{MMYY}-CONC-QM (e.g. DF 01-S0526-FLOR-200M) · regenerated on claim with the claiming designer's letter.
Notifications: task assigned → designer; task done → designer + admins + coordinators; task completed → same; pool claimed → previous assignee (if any).

↓
Stage 4: Final Approval
Admin / coordinator grants final approval
Revision Requested
Designer fixes + re-submits (tracked in completion_history)
↓
Re-submit
Loops back to review
Reject
Terminal state
↓
Activity Timeline
All events logged: submissions, reviews, edits, status changes, hold/resume
Monthly target: 3 approved concepts per designer
Auto-reminders: useConceptReminders fires on Day 8 (≥1), Day 17 (≥2), Day 24 (≥3) — sends warning notification if designer is behind target
Revision tracking: completion_history JSONB column records each revision cycle with timestamps + delay_days + notes
Multi-file: concepts.files JSONB array supports multiple file uploads per concept (migration 0018)
Notifications: Concept submitted → admins, reviewed → designer, re-submitted → admins
Stage 2 restriction: Only admin and super_admin can perform the MD review (Stage 2). Coordinators have Stage 4 final approval access but NOT Stage 2.
Concept editing: Designers can edit their own concepts (description, files, metadata) until MD approves. Uses SubmitConceptDialog in edit mode.
Activity timeline: All concept events are logged and displayed in the detail drawer — submissions, reviews, edits, hold/resume actions.
Dashboards: Concept analytics tab (funnel + leaderboard + target race + MD review panel)

6. Full Kitting Workflow (2-Stage Handover)

Stage A Coordinator

Coordinator photographs the paper kitting form and uploads via KittingStageADialog (from task row action menu on /dashboard).

File lands in sample-files/{uid}/kitting/....
Creates full_kitting_details row with image_url set and data_entry_status = 'pending_deo'.

Notifies all DEOs

→

Stage B DEO Digitizes

DEO opens /kitting queue → clicks "Open form" → /kitting/:recordId.

Side-by-side layout: paper form photo (sticky) + 12-field digital form (Fabric, Width, Design Count, Type, Theme, Background, Garment, Motive Size, Concept, APC, Additional, Priority).

Draft auto-saves to localStorage. On submit, DB trigger auto-flips status → 'completed'.

Notifies admin + coordinator

→

Stage C Coordinator Review

Coordinator re-opens the task's Full Kitting menu item.

Dialog shows status = Completed with thumbnail + "Open digital form" link to the populated form.

Completed forms appear in the CompletedKittingPanel (searchable + CSV export).

Enums (0021): kitting_data_entry_status (pending_image / pending_deo / in_progress / completed) · kitting_priority (very_urgent / 2_days / 3_days / 4_days / 5_days)
Priority mapping: lib/kitting.ts translates between display strings ("Very Urgent") and DB enums (very_urgent)
DEO landing: roleHomePath('deo') returns /kitting. DEO sidebar has only Kitting Queue + Notifications.
