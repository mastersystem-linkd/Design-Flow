3. Role Permissions (5 Roles)

Super Admin Highest Level

  • All Admin powers + Danger Zone access
  • Stage 2 concept approval (MD review)
  • Full system configuration
  • User role management + designer codes
  • Scorecards (grid + detail views)
  • CSV Export on all views
  • Sampling, Salvedge, Files, Team

Admin Full Power

  • All Tasks, Task Dashboard, Concept Dashboard
  • Stage 2 concept approval (admin/super_admin only)
  • Change user roles + manage designer codes
  • Manage lookup tables (System CRUD)
  • Scorecards (grid + detail views)
  • CSV Export on all views
  • Sampling, Salvedge, Files, Team

Coordinator Admin-Equivalent

  • Same as Admin (permissions now equal)
  • Task lifecycle (create, assign, advance)
  • Coordinator Tasks page (/coordinator-tasks)
  • Stage 4 final approval access (NOT Stage 2)
  • User management (add/edit/deactivate)
  • System CRUD access
  • Sampling, Salvedge, Files, Team
  • No Scorecards page access

Designer Submit & Work

  • My Board, Task Dashboard, Concepts, Files
  • Claim tasks from open pool
  • Submit + finalize concepts
  • Can edit own concepts until MD approves
  • Hold / Resume concepts
  • Own scorecard view (/scorecards/:self)
  • Salvedge (own assigned records only)
  • Cannot create briefs or manage team
  • No Sampling, System access

DEO Kitting Only

  • Kitting Queue (/kitting) is their landing page
  • Digitizes paper kitting forms (12-field)
  • Can claim + complete kitting records
  • Profile + Notifications only
  • Cannot create tasks or briefs
  • No dashboards, concepts, files, team

4. Task Pipeline

Pool
Unassigned · FIFO claim
In Progress
Designer working
Done
Awaiting fabric
Completed
Terminal (fabric recorded)
Full Kitting
Side data tab (not a pipeline stage) · 2-stage coordinator→DEO workflow on the row's action menu
Sampling
Own page at /sampling · runs in parallel, not a downstream stage
Stepper UI: <TaskPipelineStepper> — slim single-row glass pills mounted as the All Tasks table header. Connected: Pool → In Progress → Completed (chevrons fill once upstream has items). Full Kitting renders as a standalone side pill divided off to the right (NOT a pipeline step) via the sideStage prop.
done vs completed (load-bearing): done = design work finished, fabric not yet captured (intermediate "awaiting completion details" state). completed = terminal once fabric is recorded. On dashboards, "Done" merges both; on the board, done tasks stay in In Progress badged "Done" with a per-row "Complete" CTA.
Pool claim flow: Designers don't browse the pool — <ClaimTaskModal> shows only the single FIFO-eligible task (urgent-first, then oldest requirement_received_at) with all brief details + reference files. They claim that one or cancel — no cherry-picking. Claim sets planned deadline (required), optional fabric, optional designType, status=in_progress, designer letter on task_code, optimistic lock on .eq('status','pool').
Pool skip + FK warning chain: openClaimOrWarn(task) → skip dialog (if not front of queue) → FK warning dialog (if FK required but missing) → ClaimTaskModal. Works from pool table, drawer, and cards.
FK Gate: isFullKittingBlocking(task) blocks markTaskDone() and completeTask() when a task requires FK but the coordinator hasn't uploaded it. Task stays in_progress until FK is added via KittingStageADialog.
FK Coordinator Workflow: On claim, flagFkPendingToCoordinator() auto-creates a coordinator to-do via create_fk_coordinator_task RPC. When coordinator adds FK, completeFkCoordinatorTask() auto-closes it via complete_fk_coordinator_task RPC.
Completion: Mark Done (FK gate check) → done + notifies designer + admins/coordinators. "Complete" opens <PostDoneModal> which collects fabric (required), designType (required), and samplingRequired toggle (Switch). completeTask stamps completion_fabric/filled_by/filled_at; when samplingRequired=true, auto-calls createPendingSample().
Sampling automation: createPendingSample(taskId, fabric, designType) inserts a sample row (source='task_completion', sample_status='pending'). Deduped by (task_id, fabric, design_type). Appears in Sampling Hub's Pending Samples tab for coordinator processing.
Return to Pool (3 modes): reset (wipe all), split-pool (preserve work, return remainder), split-assign (preserve work, assign remainder to specific designer).
Handoff: handoffTask() transfers task between designers preserving qty_completed progress.
qty_completed may EXCEED qty: the inline column CHECK referenced two columns and Postgres auto-named it tasks_check (NOT tasks_qty_completed_check). 0043 dropped the wrong name; 0044 drops the real one and re-asserts only qty_completed >= 0. QtyTracker clamp has no upper bound.
Realtime: tasks joined the supabase_realtime publication (0041, REPLICA IDENTITY FULL); pool claims/assignments propagate live (~1s). concepts followed (0054).
Task code: DF NN-D{MMYY}-CONC-QM (e.g. DF 01-S0526-FLOR-200M) · regenerated on claim with the claiming designer's letter.
Notifications: task assigned → designer; task done → designer + admins + coordinators; task completed → same; pool claimed → previous assignee (if any).

12. Sampling Automation Flow

Designer completes task
PostDoneModal: fabric (required) + designType (required) + samplingRequired toggle (Switch)
↓
completeTask()
FK gate check → status=completed → stamps completion fields → if samplingRequired: createPendingSample()
↓
createPendingSample()
Dedup: unique index (task_id, fabric, design_type) · source='task_completion' · sample_status='pending' · party via 2 queries (LD briefs → default LD party from clients, never hardcoded) · toasts on real failure
↓
Pending Samples Tab
PendingSamplesPanel in Sampling Hub · coordinator clicks "Process" → pre-filled SamplingFormDialog · or "Delete"
DB columns (0069-0070): tasks: sampling_required, sampling_flagged_at, sampling_flagged_by. samples: sample_status, source, design_type.
Party resolution: two plain queries (task → client), NOT a nested embed (the embed returns a null client → blank party). LD briefs (no client_id) resolve their party from the clients table (LD group "LD Silk Mills") via resolveDefaultLdParty() — backend-driven, never a hardcoded string.
Schema-cache gotcha: 0069/0070 add the samples columns but don't NOTIFY pgrst, 'reload schema'. If pending samples silently stop appearing, reload the cache — the insert was failing on an unknown column (now surfaced via toast).
Flag-later (role-safe): a completed task's ⋮ menu "Mark Sampling Required" sets the flags + creates the sample. The resulting "Sampling Flagged ✓" item navigates to /sampling for admin/coord, but is a non-clickable confirmation for designers (they can't open /sampling).
Main samples tab: uses excludePendingTaskSamples=true to hide pending auto-created samples (they live only in the Pending tab).
Split tasks: each portion can trigger its own pending sample with different fabric/designType combinations.

13. FK Coordinator Workflow

Designer ACTUALLY claims FK-required task
Fires from ClaimTaskModal onClaimed (real claim only) · gated on isFullKittingBlocking()
↓
flagFkPendingToCoordinator(taskId, code, designer)
create_fk_coordinator_task RPC (SECURITY DEFINER) · deduped one per task · NEVER on the "Continue Without FK" intent — backing out of the form sends nothing
↓
Coordinator to-do + "Add FK ↗" redirect
coordinator_tasks row w/ related_task_id · row button → /dashboard?status=in_progress&focus=<id> · KanbanView hard-filters to that task + focus banner
↓
Coordinator uploads FK
KittingStageADialog → full_kitting_details row created → isFullKittingBlocking() = false
↓
completeFkCoordinatorTask()
complete_fk_coordinator_task RPC · auto-closes the to-do (Pending → Done) · designer notified, can now complete
Timing (load-bearing): the to-do fires ONLY from the claim's onClaimed (a committed claim), never on the "Continue Without Full Knitting" click — so a designer who opens then closes the form spams nobody, and the to-do names whoever actually claimed (designer A starts, designer B claims → coordinator hears about B).
Add FK redirect: the to-do row's "Add FK ↗" button (shown when related_task_id && !is_completed) → /dashboard?status=in_progress&focus=<related_task_id>. KanbanView hard-filters the visible rows to that one task at the render level (stepper counts stay the true totals) + a focus banner with "Add Full Knitting" + "Clear focus".
FK Gate: isFullKittingBlocking(task) blocks both markTaskDone() and completeTask() until FK details are added.
ClaimTaskModal: shows yellow FK warning banner when task needs FK but doesn't have it (non-blocking — designer can still claim).
DB (0071-0072): coordinator_tasks.related_task_id FK (ON DELETE SET NULL) · create_fk_coordinator_task() (0072 = 3-arg) + complete_fk_coordinator_task() RPCs.
